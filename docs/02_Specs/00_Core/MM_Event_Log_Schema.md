## 1) Scope
This schema covers **MM event logs** (engine intent + broker-confirmed outcomes):
- ENTRY / SCALE_OUT / BE / TRAIL
- EXIT (intent)
- CLOSE (outcome)

This schema does NOT cover:
- Broker execution telemetry (OrderSend/OrderModify responses, raw trade transaction payloads)
→ Covered by EXEC-LOG-01

---

## 2) Canonical Event Types
Event `event_type` MUST be one of:
- MM_EVENT_ENTRY
- MM_EVENT_SCALE_OUT
- MM_EVENT_BE
- MM_EVENT_TRAIL
- MM_EVENT_EXIT   *(Intent — engine close request; optional per lifecycle)*
- MM_EVENT_CLOSE  *(Outcome — broker-confirmed closure; mandatory lifecycle terminator)*

## 3) CSV Schema (Column Order is Fixed)

### 3.1 Column List (v2.1, in order)

All Event logs MUST follow this exact column order. Missing or extra columns invalidate the row for schema compliance.

1. `debug_event_id` (ulong)
2. `correlation_id` (ulong)
3. `event_time` (string datetime)
4. `symbol` (string)
5. `timeframe` (string enum)
6. `phase` (string enum)
7. `event_type` (string enum)
8. `cycle_id` (int)
9. `trade_id` (long)
10. `ticket` (ulong)
11. `position_type` (string)
12. `action_summary` (string)
13. `scale_steps` (int)
14. `scale_fraction_total` (double)
15. `close_reason` (string enum)
16. `close_price` (double)
17. `close_profit` (double)
18. `close_volume` (double)
19. `deal_id` (long)

### 3.2 Core Field Notes

- `debug_event_id` is a monotonic debug sequence ID generated by the logger.
- `correlation_id` is the action-level trace identifier for one logical MM action/evaluation chain inside a cycle_id. For successful non-CLOSE actions, it binds the Event row to the corresponding Snapshot BEFORE/AFTER pair. For CLOSE events, correlation behavior follows the CLOSE correlation rules in Section 4.5.
- `event_time` is the event timestamp emitted by the EA.
- `timeframe` MUST use `EnumToString` output, e.g. `PERIOD_M15`.
- `phase` MUST use canonical MM phase strings, e.g. `MM_PHASE_ENTRY`, `MM_PHASE_MANAGE`, `MM_PHASE_EXIT`.
- `event_type` MUST match one of the canonical Event types in Section 2.
- `cycle_id` is the lifecycle grouping ID for ENTRY → ... → CLOSE reconstruction.
- `trade_id` is the internal/logging trade identifier. In the current implementation, it is aligned with broker ticket where applicable.
- `ticket` is the broker ticket.
- `position_type` is the trade direction context for the Event row.
- `action_summary` is a human-readable action description.
- `scale_steps` and `scale_fraction_total` apply to SCALE_OUT and are `0` / `0.0` otherwise.
- `close_reason`, `close_price`, `close_profit`, `close_volume`, and `deal_id` are broker-confirmed close/partial-close evidence fields.
- `close_volume` represents the volume closed in the specific event.

  **Semantics:**
  - For `MM_EVENT_SCALE_OUT` → partial volume closed during the scale-out execution
  - For `MM_EVENT_CLOSE` → remaining volume closed during final lifecycle termination

  **Invariant:**
  SUM(`close_volume` across all events within the same `cycle_id`)
  equals the total lifecycle closed volume (Cycle Summary close_volume)

### 3.3 position_type Rule (v2.1)

`position_type` provides direction context at event time.

**Allowed values:**

- `LONG`
- `SHORT`
- `NA`

**Rules:**

- If a live position is selectable at event time, `position_type` MUST reflect live position direction.
- ENTRY events MAY use intended entry direction before broker confirmation.
- CLOSE events MAY use last-known cached position direction.
- If direction cannot be derived, `position_type` MUST be `NA`.
- Trend values such as `UpTrend` or `DownTrend` are not valid for `position_type`.

## 4) Field Rules (Strict)

### 4.1 Required vs Optional

- Columns 1–19 MUST exist for every Event row.
- Core identity/classification fields MUST be populated for every Event row:
  - `debug_event_id`
  - `correlation_id`
  - `event_time`
  - `symbol`
  - `timeframe`
  - `phase`
  - `event_type`
  - `cycle_id`
  - `trade_id`
  - `ticket`
  - `position_type`
  - `action_summary`
- `position_type` MUST be one of:
  - `LONG`
  - `SHORT`
  - `NA`
- SCALE_OUT fields:
  - `scale_steps` MUST be populated.
  - `scale_fraction_total` MUST be populated.
  - For non-SCALE_OUT events, `scale_steps = 0` and `scale_fraction_total = 0.0`.
- CLOSE evidence fields:
  - MUST be populated when `event_type = MM_EVENT_CLOSE`.
  - MAY be populated when `event_type = MM_EVENT_SCALE_OUT` and a broker partial-close deal is matched.
  - MUST remain neutral for ENTRY / BE / TRAIL / EXIT.
- close_volume semantic rules (v2.2):
  - MUST be populated (> 0) for:
    - MM_EVENT_SCALE_OUT (when a partial close is executed)
    - MM_EVENT_CLOSE (final close)
  - MUST be 0 or empty for:
    - `MM_EVENT_ENTRY`
    - `MM_EVENT_BE`
    - `MM_EVENT_TRAIL`
    - `MM_EVENT_EXIT`
  - MUST represent actual executed volume per event, based on broker-confirmed deal data.
  - Aggregation rule:
    SUM(`close_volume` across all events in the same `cycle_id`) MUST equal the total lifecycle closed volume.

### 4.2 close_reason (required for CLOSE)
`close_reason` MUST be one of:
- MM_EXPERT
- MM_EXPERT: Exit Signal
- MM_EXPERT: Scale Out
- MANUAL_DESKTOP_TERMINAL
- MANUAL_MOBILE_APP
- MANUAL_WEB_PLATFORM
- TP_HIT
- SL_HIT
- STOP_OUT Event
- ROLLOVER
- VARIATION_MARGIN
- CORPORATE_ACTION
- SPLIT_ANNOUNCEMENT
- UNKNOWN


### 4.3 Two-phase termination rules (MM-LOG-01 alignment)
- A lifecycle MAY contain `0..1` `MM_EVENT_EXIT` rows.
- `MM_EVENT_EXIT` is intent-only and represents an engine close request.
- A completed lifecycle MUST contain exactly one `MM_EVENT_CLOSE`.
- `MM_EVENT_CLOSE` is the broker-confirmed lifecycle terminator.
- EXIT may be absent for broker-driven closures such as TP/SL.

### 4.4 Consistency requirements
- `cycle_id` MUST be constant across all events belonging to the same lifecycle.
- `cycle_id` MUST increment on each ENTRY lifecycle start.
- `correlation_id` MUST allow joining successful non-CLOSE Event rows to corresponding Snapshot BEFORE/AFTER rows.
- For successful non-CLOSE actions:
  - Event `event_type` MUST match Snapshot `mm_event`.
  - Event `ticket` MUST match Snapshot `ticket`.
  - Event `position_type` MUST match Snapshot `position_type`.
- `ticket` and/or `trade_id` MUST allow joining across logs when available.
- `phase` and `timeframe` MUST use `EnumToString` output, not custom shortened labels.
- CLOSE is Event-only broker confirmation under the current v2.1 runtime model and does not require Snapshot BEFORE/AFTER rows.
- Volume consistency (v2.2 model):

  Event-level `close_volume` fields form the single source of truth for lifecycle volume.

  The Cycle Summary `close_volume` MUST be derived as:
  SUM(Event.`close_volume` for all `MM_EVENT_SCALE_OUT` and `MM_EVENT_CLOSE` events within the same `cycle_id`).

  No separate lifecycle volume field is maintained.

### 4.5 correlation_id rules for MM_EVENT_CLOSE

correlation_id is the action-level trace identifier used to group logs belonging to one logical MM action/evaluation chain.

It is intentionally different from cycle_id:

- cycle_id groups the full trade lifecycle.
- correlation_id groups one logical action/evaluation chain inside that lifecycle.
- debug_event_id identifies one physical Event log row.

MM_EVENT_CLOSE is emitted by close detection after the engine observes that a previously open position is no longer open. Under the current v2.1 runtime model, CLOSE is Event-only broker confirmation and does not require Snapshot BEFORE/AFTER rows.

##### Engine-driven EXIT → CLOSE correlation

When MM_EVENT_CLOSE is the broker-confirmed result of an explicit engine exit action, the MM_EVENT_CLOSE row MAY preserve the same correlation_id as the originating MM_EVENT_EXIT row.

This shared correlation_id pattern is valid only when:

```text
MM_EVENT_EXIT → MM_EVENT_CLOSE
close_reason = MM_EXPERT: Exit Signal
```

## 5) JSON Event Object Schema

### 5.1 Required keys (all events)

Each JSON event MUST contain keys matching the CSV field names:

- `debug_event_id`
- `correlation_id`
- `event_time`
- `symbol`
- `timeframe`
- `phase`
- `event_type`
- `cycle_id`
- `trade_id`
- `ticket`
- `position_type`
- `action_summary`
- `scale_steps`
- `scale_fraction_total`
- `close_reason`
- `close_price`
- `close_profit`
- `close_volume`
- `deal_id`

### 5.2 Required keys for MM_EVENT_CLOSE and MM_EVENT_SCALE_OUT

For MM_EVENT_CLOSE:
- `close_reason` / `close_price` / `close_profit` / `close_volume` / `deal_id` MUST be populated.

For MM_EVENT_SCALE_OUT:
- `close_reason` / `close_price` / `close_profit` / `close_volume` / `deal_id` MUST be populated when a broker partial-close deal is matched.
- `close_volume` MUST represent the actual executed partial-close volume for that event.

For all other event types:
- `close_reason` / `close_price` / `close_profit` / `close_volume` / `deal_id` MUST be null/empty.
